# Escalation Ladder

使用此文件防止跳级——当你在当前阶遇到阻力时，确认穷尽了当前手段再升级。
这个 skill 的核心就是"能简单就不上工具"，阶梯就是保证这条信念落地的机制。
每一阶都有准入条件和退出条件，此外还有两个**穿插能力**（结构化侦察、验证码流水线）贯穿全程。
验证码不是升级信号——遇到验证码优先走 captcha.md 流水线，而不是跳阶。

## 核心规则

**不跳级。** 在升级到下一阶之前，必须记录：

1. 当前阶仍然生效的最新证据
2. 当前阶的精确失败点
3. 为什么下一阶是最小的可解决问题的层级
4. 为什么升级后交付物仍然不依赖浏览器
5. 是否已尝试过验证码流水线（captcha.md）——如果目标是验证码页面

如果写不出这些（至少前四条），不能升级。

## SpiderMind 五阶阶梯

### 阶 0：结构化侦察（Recon）

证明：
- 站点架构已识别（SSR/SPA/Vue/React）
- 反爬指纹已分类（IP封/JS Challenge/加密参数/验证码/Cloudflare/WASM/字体反爬）
- 请求链路已追踪（Cookie → Header → Token 的依赖关系）
- 数据画像已完成（JSON/HTML/加密文本，分页方式，数据量预估）
- 侦察报告已输出（参见 recon.md 模板）

留在这里当：
- 站点类型还不确定
- 响应格式还没搞清楚
- 反爬类型还有疑点
- 数据来源还没确认

升级条件（进入一阶）：
- 侦察报告完整，站点画像清晰
- 已明确至少知道目标数据从哪个 URL 来
- 已明确主要反爬手段是什么（哪怕只是一个猜测）

### 阶 1：确认基线

证明：
- 目标 URL 可访问
- 已知反爬类型
- 已知响应格式
- 项目骨架已创建

留在这里当：
- 目标还没理解透
- 反爬类型还没分类

升级条件：基线清晰、项目骨架就绪

### 阶 2：裸 Py 探路

证明：
- 裸请求的响应状态码和内容
- 逐步加 headers 后哪个是关键的
- Cookie/session 是否需要及如何获取
- 是否触发了验证码（如触发，启动 captcha.md 流水线而非跳阶）

留在这里当：
- 每次加一个 header 还有进展
- 响应数据只是简单编码（base64/hex）
- 换个 IP 就能解决
- 出现验证码但 ddddocr 流水线能搞定

升级条件（进入三阶）：
- headers/cookies 都加了仍被拦，且拦截特征是 JS 动态参数
- 响应数据被 JS 加密，解码逻辑在 JS 中
- 出现 JS 挑战（如 __jsl_clearance 等）
- 验证码流水线（captcha.md）重试 3 次仍失败且无法降级绕过

### 阶 3：补环境扣代码

证明：
- Playwright 定位到了加密参数的"出生点"（具体 JS 文件和函数）
- 扣下来的代码在 Node.js 中能加载（至少不报 ReferenceError）
- 固定输入下 Node.js 输出与浏览器一致或差异可解释
- 过程中如遇验证码，通过 captcha.md 流水线处理

留在这里当：
- 加密代码找到了，正在补环境
- 输出接近但不完全一致，差异可以逐个修复
- 混淆不严重，手动可读
- 补环境过程中经常触发验证码，但 captcha.md 能稳定处理

升级条件（进入四阶）：
- 代码 VMP 虚拟化，控制流无法理解
- 代码严重混淆且加密入口找不到
- WASM 签名
- 补环境后输出始终有差异且无法定位差距

### 阶 4：jsreverser 深度逆向

证明：
- jsreverser-mcp 已启动并加载目标 JS 文件
- 已输出还原后的逻辑或最小可运行片段
- 固定输入下输出与浏览器一致

留在这里当：
- jsreverser 正在分析中
- 还原结果需要多轮验证

降级条件（回到三阶）：
- 发现代码其实可以手动读懂，不需要 jsreverser
- 发现是标准库（CryptoJS 等）没有魔改，可直接 npm 安装

### 升级检查单

在升级前逐项确认：

使用以下格式记录每次阶位变动：

| 记录项 | 内容 |
|--------|------|
| 当前阶最新有效证据 | 什么请求能跑通 |
| 当前阶精确失败点 | 状态码 + 错误信息 |
| 为什么下一阶是最小方案 | 为什么不能更轻 |
| 升级后交付是否仍不依赖浏览器 | 是/否 |
| 是否已尝试验证码流水线 | 是/否，如已尝试则附结果 |

## 降级规则

如果发现升级是误判：

| 原台阶 | 降级信号 | 操作 |
|--------|----------|------|
| 阶 3 | 发现请求只需要正确的 headers + cookie | 回阶 2，直接 requests |
| 阶 3 | cookie 可以从登录接口用 requests 获取 | 回阶 2，requests 模拟登录 |
| 阶 4 | 加密代码在某个清晰的 JS 文件中，手动能读懂 | 回阶 3，直接扣代码 |
| 阶 4 | 发现是标准库（CryptoJS 等）没有魔改 | 回阶 3，npm 安装标准库 |
| 任意阶 | 遇到验证码 | 不走降级，走 captcha.md 流水线 |

降级不需要羞愧——降级说明你之前的分析在逼近真相。

## 禁止的跳级行为

- 不做结构化侦察（recon.md）就直接开始写代码
- headers 没加全就说"这个站很复杂"直接上 Playwright
- Playwright 还没看清 Network 面板就说"搞不定"直接 jsreverser
- 补环境报了第一个错就说"环境差异太大"放弃三阶
- 把浏览器卡顿、网络慢当成"JS 挑战"进入三阶
- 把 403 当成"必须上 jsreverser"——可能只是 IP 被拉黑
- 把 base64 编码当成"响应加密"进入三阶——data:image 后面也是 base64
- 不写升级记录就悄悄开始下一阶
- 遇到验证码就直接跳阶——正确做法是先走 captcha.md 流水线
- 把验证码页当成"这个站很复杂"的升级理由

## 阶梯日志格式

在每次阶位变动时，用这个格式记录：

```markdown
### 阶梯日志

**目标**: {目标 URL}
**当前阶**: {0/1/2/3/4}
**上次验证**: {时间，什么请求能跑通}
**当前失败**: {精确的失败现象，含状态码和错误信息}
**尝试过的方案**: {列举尝试过的手段}
**验证码尝试**: {无 / 已用 captcha.md 流水线处理 / ddddocr 识别成功 / 3次重试失败}
**升级/降级决定**: {阶 X → 阶 Y}
**理由**: {为什么需要/可以换阶}
**交付形态**: {纯 Python / Python + Node helper / 待定}
```

## 底线

阶梯存在的唯一目的：**让每一步都有退路，让每一次升级都有证据，让每一个验证码都有流水线可走。**

能在零阶想清楚的不要上二阶，能在一阶搞定的不要上三阶，能在三阶搞定的不要上四阶。
什么时候上四阶？当前三阶的证据明确表明"此路不通"的时候。
什么时候走验证码流水线？任何时候遇到验证码都可以走——但验证码不是升级的理由。
