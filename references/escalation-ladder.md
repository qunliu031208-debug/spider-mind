# Escalation Ladder

使用此文件防止跳级——当你在当前阶遇到阻力时，确认穷尽了当前手段再升级。
这个 skill 的核心就是"能简单就不上工具"，阶梯就是保证这条信念落地的机制。

## 核心规则

**不跳级。** 在升级到下一阶之前，必须记录：

1. 当前阶仍然生效的最新证据
2. 当前阶的精确失败点
3. 为什么下一阶是最小的可解决问题的层级
4. 为什么升级后交付物仍然不依赖浏览器

如果写不出这四条，不能升级。

## SpiderMind 三阶阶梯

### 阶 0：确认基线

证明：
- 目标 URL 可访问
- 已知反爬类型
- 已知响应格式
- 项目骨架已创建

留在这里当：
- 目标还没理解透
- 反爬类型还没分类

升级条件：基线清晰、项目骨架就绪

### 阶 1：裸 Py 探路

证明：
- 裸请求的响应状态码和内容
- 逐步加 headers 后哪个是关键的
- Cookie/session 是否需要及如何获取

留在这里当：
- 每次加一个 header 还有进展
- 响应数据只是简单编码（base64/hex）
- 换个 IP 就能解决

升级条件（进入二阶）：
- headers/cookies 都加了仍被拦，且拦截特征是 JS 动态参数
- 响应数据被 JS 加密，解码逻辑在 JS 中
- 出现验证码页面或 JS 挑战

### 阶 2：补环境扣代码

证明：
- Playwright 定位到了加密参数的"出生点"（具体 JS 文件和函数）
- 扣下来的代码在 Node.js 中能加载（至少不报 ReferenceError）
- 固定输入下 Node.js 输出与浏览器一致或差异可解释

留在这里当：
- 加密代码找到了，正在补环境
- 输出接近但不完全一致，差异可以逐个修复
- 混淆不严重，手动可读

升级条件（进入三阶）：
- 代码 VMP 虚拟化，控制流无法理解
- 代码严重混淆且加密入口找不到
- WASM 签名
- 补环境后输出始终有差异且无法定位差距

### 升级检查单

在升级前逐项确认：

```markdown
## 升级记录：阶 X → 阶 X+1
- 当前阶最新有效证据：
- 当前阶精确失败点：
- 为什么下一阶是最小方案：
- 升级后交付是否仍不依赖浏览器：
```

## 降级规则

如果发现升级是误判：

| 原台阶 | 降级信号 | 操作 |
|--------|----------|------|
| 阶 2 | 发现请求只需要正确的 headers + cookie | 回阶 1，直接 requests |
| 阶 2 | cookie 可以从登录接口用 requests 获取 | 回阶 1，requests 模拟登录 |
| 阶 3 | 加密代码在某个清晰的 JS 文件中，手动能读懂 | 回阶 2，直接扣代码 |
| 阶 3 | 发现是标准库（CryptoJS 等）没有魔改 | 回阶 2，npm 安装标准库 |

降级不需要羞愧——降级说明你之前的分析在逼近真相。

## 禁止的跳级行为

- headers 没加全就说"这个站很复杂"直接上 Playwright
- Playwright 还没看清 Network 面板就说"搞不定"直接 jsreverser
- 补环境报了第一个错就说"环境差异太大"放弃二阶
- 把浏览器卡顿、网络慢当成"JS 挑战"进入二阶
- 把 403 当成"必须上 jsreverser"——可能只是 IP 被拉黑
- 把 base64 编码当成"响应加密"进入二阶——data:image 后面也是 base64
- 不写升级记录就悄悄开始下一阶

## 阶梯日志格式

在每次阶位变动时，用这个格式记录：

```markdown
### 阶梯日志

**目标**: {目标 URL}
**当前阶**: {0/1/2/3}
**上次验证**: {时间，什么请求能跑通}
**当前失败**: {精确的失败现象，含状态码和错误信息}
**尝试过的方案**: {列举尝试过的手段}
**升级/降级决定**: {阶 X → 阶 Y}
**理由**: {为什么需要/可以换阶}
**交付形态**: {纯 Python / Python + Node helper / 待定}
```

## 底线

阶梯存在的唯一目的：**让每一步都有退路，让每一次升级都有证据。**

能在一阶搞定的不要上二阶，能在二阶搞定的不要上三阶。
什么时候上三阶？当前两阶的证据明确表明"此路不通"的时候。
